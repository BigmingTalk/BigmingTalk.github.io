虚拟存储器之物理寻址、虚拟寻址及分页

计算机系统
  • MMU
  • TLB
  • 分页
  • 物理寻址
  • 虚拟存储器
  • 虚拟寻址

虚拟存储器（VM）为主存提供了一种抽象，其是硬件异常、硬件地址翻译、主存、磁盘文件和内核软件的完美交互，它为每个进程提供了一个大的、一致的和私有的地址空间。虚拟存储器主要提供了下面三个重要的功能：

  1. 将主存看成是一个存储在磁盘上的地址空间的高速缓存，在主存中只保存活动区域，并根据需要在磁盘和主存之间来回传送数据，因此高效地使用了主存。
  2. 为每个进程提供了一致的地址空间，从而简化了存储器管理。
  3. 保护了每个进程的地址空间不被其他进程破坏。

物理寻址和虚拟寻址

物理寻址

计算机系统主存被组织为一个由M个连续字节大小的单元组成的数组。每个字节都有一个唯一的物理地址（Physical Address, PA），第一个字节地址为0，最后一个地址为M-1。如果CPU访问存储器采用的是物理地址，我们称之为物理寻址。下图就是一个简单的物理寻址的示例，其读取物理地址4上面的一个字（4字节）。

早期的PC，现在的数字信号处理器，嵌入式微控制器仍然使用该种寻址方式。在下面的讨论中，物理存储器，DRAM，内存以及主存指的是同一个意思。

虚拟寻址

虚拟寻址就是CPU先生成一个虚拟地址（Virtual Address, VA）来访问主存，该虚拟地址在被送到存储器之前先转换成适当的物理地址。在下面讨论中，虚拟存储器指的就是磁盘。


地址翻译就是将一个虚拟地址转换为物理地址的过程。和异常处理异议，地址翻译需要CPU硬件和操作系统合作完成。CPU芯片上叫做存储器管理单元（Memory Management Unit, MMU）的专用硬件，利用存放在主存中的查询表来动态翻译虚拟地址，该表的内容由操作系统管理。

地址空间

地址空间是一个非负整数地址的有序集合：{0, 1, 2, …}。如果地址空间中的整数是连续的，那么我们就说它是一个线性地址空间。一般我们总是假设使用的是线性地址空间。CPU从一个N=2^n个地址的地址空间中生成虚拟地址，这个地址空间称为虚拟地址空间，即{0, 1, 2,…, N-1}

一个地址空间的大小由表示最大地址所需要的位数来描述。例如，一个包含N=2^n个地址的虚拟地址空间就叫做一个n位地址空间。现代系统一般都是32位或者64位地址空间。

物理地址空间是与系统中物理存储器的M个字节相对应：{0, 1, 2,…, M-1}。M不要求是2的幂，不过一般简化讨论，也都假设M=2^m。

地址空间区分了数据对象（字节）和它们的属性（地址），因此每个数据对象可以有多个独立的地址，每个地址都来自一个不同的地址空间，这就是虚拟存储器的基本思想。主存中每个字节都有一个选自虚拟地址空间的虚拟地址和一个选自物理地址空间的物理地址。

虚拟存储器作为缓存的工具

概念上来说，虚拟存储器被组织成一个存放在磁盘上N个连续字节大小的单元的数组。每字节都有一个唯一的虚拟地址作为到数组的索引。磁盘上的数组内容被缓存在主存中。和存储器层次结构中其他缓存一样，磁盘上的数据被分割成块，这些块作为磁盘和主存之间的传输单元。VM系统将虚拟存储器分割成称为虚拟页的大小固定的块来处理，每个虚拟页的大小为P=2^p字节。类似地，物理存储器被分割为物理页，大小也为P字节（物理页也称为页帧）。

虚拟页面的集合分为三个不相交的子集：

  • 未分配的：VM系统还未分配（或者创建的）页。未分配的块没有任何数据和它们相关联，因此不占用任何磁盘空间。
  • 缓存的：当前缓存在主存中的已分配页。
  • 未缓存的：没有缓存在主存中的已分配页

DRAM缓存的组织结构

SRAM缓存表示位于CPU和主存之间的L1,L2,L3高速缓存，DRAM缓存表示虚拟存储器系统的缓存，它在主存中缓存虚拟页。

一般来说，SRAM比DRAM块大约10倍，而DRAM比磁盘快100000多倍。因此DRAM缓存中的不命中比SRAM缓存中的不命中代价要高很多。因此，虚拟页往往很大以弥补不命中的开销，典型的是4KB~2MB。并且DRAM缓存是全相联的（关于全相联可以查看深入浅出理解高速缓存及多路组相联）。 也就是任何虚拟页都可以放置在任何的物理页中。不命中的替换算法也非常重要，并且因为对磁盘的访问时间很长，DRAM缓存总是使用写回，而不是直写。

页表

虚拟存储器需要判断一个虚拟页是否在DRAM中，如果是的话还需要确定虚拟页存放在哪个物理页中。如果不命中需要判断该虚拟页放在磁盘的哪个位置，在主存中选择一个牺牲页，并将虚拟页从磁盘拷贝到DRAM中，替换该牺牲页。这些功能由操作系统，MMU中的地址翻译硬件和一个存放在主存的页表数据结构合作完成。

页表将虚拟页映射到物理页，每次地址翻译硬件将一个虚拟地址转换为物理地址时都会读取页表。操作系统负责维护页表内容，以及在磁盘和DRAM之间传送页。

页表有多个页表条目（Page Table Entry, PTE），虚拟地址空间中每个页在页表中一个固定偏移量处都有一个PTE。每个PTE由一个有效位和一个n位地址段组成。有效位表明该虚拟页是否缓存在DRAM中，如果有效位为1则地址字段表示DRAM中相应的物理页的起始位置，该物理页缓存了虚拟页。如果没有设置有效位（为0），则一个空地址表示该虚拟页未被分配，否则该地址指向虚拟页在磁盘上的起始位置。下图是一个页表的基本结构：

缺页

一般来说，DRAM缓存不命中称为缺页。下图是缺页之前的一个状态。

CPU引用VP 3中的一个字，VP 3未被缓存在DRAM中，地址翻译硬件读取PTE 3，从有效位判断出VP 3未被缓存，触发一个缺页异常。

缺页异常调用内核中的缺页异常处理程序（冠以异常控制流可以参考我的另一篇文章操作系统之进程及异常控制流）。 该程序会选择一个牺牲页，此例中为存放在PP 3中的VP 4.如果VP 4已结被修改了，则内核将其拷贝回磁盘。并且无论如何内核都会修改VP 4的页表条目，反映出VP 4不再缓存在主存中的这一事实。

然后内核从磁盘拷贝VP 3到DRAM的PP 3，更新PTE 3，随后返回。当异常处理程序返回时，它会重新启动导致缺页的指令，该指令会导致缺页的虚拟地址重发送到地址翻译硬件，此时VP 3已结缓存在DRAM中了。下图是缺页之后的页表状态。

虚拟存储器最早在20世纪60年代发明，因此使用了与SRAM缓存不一样的术语，虽然它们很多概念是相同的。比如在虚拟存储器中块被称为页。在磁盘和主存之间传送页称为交换或者页面调度。页从磁盘换入（页面调入）DRAM和从DRAM换出（页面换出）磁盘。一直等待，直到最后时刻也就是有不命中发生时，才换入页面的策略叫做按需页面调度，当然也可以尝试预测不命中，在页面实际被引用之前就换入页面。不过，所有现代系统使用的都是按需页面调度完成的。

页面分配


上图展示了操作系统分配一个新的虚拟存储器页时对我们示例页表的影响，这通常是调用malloc函数的结果。通过在磁盘上创建空间并更新PTE 5，使它指向磁盘上这个新创建的页面，从而分配VP 5。

局部性

因为不命中处罚很大，感觉因此造成性能会很低，但是虚拟存储器实际上因为局部性的原因性能还是很不错的。尽管在运行过程中程序引用的不同页面的总数可能超出内存的大小，但是局部性保证了在任意时刻，程序往往在一个较小的活动页面集合上工作，该集合叫做工作集或者常驻集。只要程序具有良好的时间局部性，虚拟存储器系统就能工作地很好。如果工作集的大小超过了主存的大小，就会导致页面不断地换进换出，称之为颠簸状态。

虚拟存储器作为存储器管理的工具

上一节中我们看到了利用DRAM缓存来自通常更大的磁盘虚拟地址空间的页面。但是其实早期DEC PDP-11/70系统支持的是一个比DRAM更小的磁盘虚拟地址空间。但是虚拟地址仍然是有用的，因为它简化了存储器管理，并提供了一种自然的保护存储器的方法。

实际上，操作系统为每个进程提供一个独立的页表，因而也就是一个独立的虚拟地址空间，如下图所示：

上图中有一个重要的信息就是多个虚拟页面可以映射到同一个物理页面上。按需页面调度和独立的虚拟地址空间的结合，简化了链接和加载、代码和数据共享，以及应用程序的存储器分配。

  • 简化链接：独立的地址空间允许每个进程的存储器映像使用相同的基本格式，而不管代码和数据实际存放在DRAM的何处。每个Linux系统上的进程都使用类似的存储器格式，文本总是从虚拟地址0x08048000处开始（32位地址空间），数据和bss节紧跟在文本节后面。栈占据进程地址空间最高的部分，并向下生长。这样的一致性极大地简化了链接器的设计和实现，运行链接器生成全链接的可执行文件，这些可执行文件是独立于DRAM中代码和数据的最终位置的。
  • 简化加载：虚拟存储使得很容易向存储器中加载可执行文件和共享对象文件。比如可执行文件的.text和.data节是连续的，要把这些节加载到一个新创建的进程中，Linux加载器分配虚拟页的一个连续的片，从地址0x08048000（32位地址空间）开始，把这些虚拟页标记为无效的（即未被缓存的），将页表条目指向目标文件中适当的位置。有趣的是加载器从不实际拷贝任何数据从磁盘到主存，在每个页初次被引用时，要么是CPU取指令时引用的，要么是一条正在执行的指令引用一个存储器位置时引用的，虚拟存储器就按需自动调入数据页。将一组连续的虚拟页映射到任意一个文件中的任意位置的表示法称作存储器映射。Unix提供了一个称为mmap的系统调用，允许应用程序自己做存储器映射。
  • 简化共享：一般而言，每个进程有自己私有的代码、数据、堆和栈，是不合其他进程共享的，这时操作系统创建页表将相应的虚拟页映射到不同的物理页面。但是进程有时也需要共享代码和数据，比如每个进程必须调用相同的操作系统内核的代码，或者是C标准库中的程序，比如printf。操作系统只需要简单地将不同进程中适当的虚拟页面映射到相同的物理页面，从而安排多个进程共享这部分代码的一个拷贝，而不是每个进程都包括单独的内核和C标准库的拷贝。
  • 简化存储器分配：虚拟存储器为向用户进程提供一个简单的分配额外存储器的机制。当一个运行在用户进程中的程序需要额外的堆空间时（malloc），操作系统只需要分配一个k个连续的虚拟存储器页面，并且将它们映射到物理存储器中任意位置的k个物理页面。由于页表，操作系统没有必要分配k个连续的物理存储器页面。页面可以随机地分散在物理存储器(DRAM)中。

虚拟存储器作为存储器保护的工具

操作系统控制对存储器系统的访问，比如：

  • 不允许用户进程修改它的只读文本段
  • 不允许用户进程读或修改任何内核中的代码和数据结构
  • 不允许用户进程读或者写其他进程的私有存储器
  • 不允许用户进程修改任何与其他进程共享的虚拟页面，除非所有的共享者都允许（比如进程间通信系统调用）

通过在页面的每个PTE上调价一些额外的许可位来控制对一个虚拟页面的访问十分简单，下图展示了这种思想：

上图中每个PTE都有三个许可位，SUP位表示进程必须运行在内核模式下才能访问该页。运行在内核模式下的进程可以访问任何页面，但是运行在用户模式中的进程只允许访问SUP为0的页面。READ和WRITE位控制对页面的读和写访问。

如果一条指令违反了这些许可条件，CPU就会触发一个一般保护故障，将控制传递给一个内核中的异常处理程序。Unix外壳一般称这种异常报告为段错误（segmentation fault）。

地址翻译

上图是地址翻译中可能涉及的所有的概念。从形式上说，地址翻译是一个N元素的虚拟地址空间（Virtual Address Space, VAS）中的元素和一个M元素的物理地址空间（Physical Address Space, PAS）中元素的映射，如下图所示：

下图展示了MMU如何利用页表来实现这种映射。CPU中有一个控制寄存器叫做页表基址寄存器（Page Table Base Register, PTBR）指向当前页表。n位的虚拟地址包含两个部分：一个p位的虚拟页面偏移（Virtual Page Offset, VPO）和一个n-p位的虚拟页号（Virtual Page Number, VPN）。MMU利用VPN来选择适当的PTE，例如VPN 0选择PTE 0，VPN 1选择PTE 1，以此类推。将页表条目中的物理页号（Physical Page Number, PPN）和虚拟地址中的VPO串联起来，就得到相应的物理地址。这里因为物理和虚拟页面都是P字节的，所以物理页面偏移（Pysical Page Offset, PPO）和VPO死相同的。

页面中

下图展示了当页命中时，CPU硬件执行的步骤。

  1. 处理器生成一个虚拟地址，并把它传送给MMU。
  2. MMU生成PTE地址，并从高速缓存/主存请求得到它
  3. 高速缓存/主存向MMU返回PTE。
  4. MMU构造物理地址，并把它传送给高速缓存/主存。
  5. 高速缓存/主存返回所请求的数据字给处理器。

缺页

页命中完全由硬件来处理，但是处理缺页要求硬件和操作系统内核写作完成。如下图所示：

  1. 处理器生成一个虚拟地址，并把它传送给MMU。
  2. MMU生成PTE地址，并从高速缓存/主存请求得到它
  3. 高速缓存/主存向MMU返回PTE。
  4. PTE有效位是零，所以MMU触发了一次异常，传递给CPU中的控制到操作系统内核中的缺页异常处理程序。
  5. 缺页处理程序确定出DRAM中的牺牲页，如果该页已经被修改了，则把它换出到磁盘。
  6. 缺页处理程序调入新的页面，并更新相应的PTE
  7. 缺页 处理程序返回到原来的进程，再次执行导致缺页的指令。CPU将引发缺页的虚拟地址重新发送给MMU，因为虚拟页面在现在缓存在DRAM中了，所有就会命中。

结合高速缓存和虚拟存储器

在任何使用虚拟存储器又使用SRAM高速缓存的系统中，都存在应该使用虚拟地址还是物理地址来访问SRAM高速缓存的问题。大多数系统是选择使用物理地址的。使用物理地址，多个进程同时在高速缓存中有存储块和共享来自相同虚拟页面的块称为很简单的事情。而且，高速缓存无需处理保护问题，因为访问权限的检查来自地址翻译过程的一部分。下图展示了一个物理寻址的高速缓存如何结合虚拟存储器，其主要的思路就在于地址翻译发生在高速缓存查找之前！！！这里还需要注意的是页表条目可以缓存，就像其他的数据字一样。

利用TLB加速地址翻译

每次CPU产生一个虚拟地址，MMU就必须查阅一个PTE，以便将虚拟地址翻译为物理地址，就会要求从存储器取一次数据，代价是几十到几百个周期。如果PTE刚好在L1中，开销就会在1个或2个周期。现代计算机系统会在MMU中包括一个关于PTE的小的缓存，称为翻译后备缓冲器（Translation Lookaside Buffer, TLB）。

TLB是一个小的、虚拟寻址的缓存，其中每一行都保存着一个由单个PTE组成的块。TLB通常有高度的相连性。如下图所示：

用于组选择和行匹配的索引和标记字段都是从虚拟地址中的虚拟页号中提取出来的，如果TLB有2^t个组，那么TLB索引是由VPN的t个最低位组成的，而TLB标记是由VPN的剩余的位组成的。下图展示了TLB命中时的步骤，因为所有地址翻译步骤都是在芯片上的MMU中执行的，因此非常快。

  • 第一步：CPU产生一个虚拟地址
  • 第二步和第三步：MMU从TLB中取出相应的PTE
  • 第四步：MMU将该虚拟地址翻译成一个物理地址，并且将它发送到高速缓存/主存中
  • 高速缓存/主存将所请求的数据字发回给CPU

当TLB不命中时，MMU必须从L1缓存中取出相应的PTE，如下图所示，新去处的PTE存放在TLB中，可能会覆盖一个已经存在的条目。

多级页表

当前我们都是假设系统每个进程只用一个单独的页表来进行地址翻译。假设我们由一个32位的地址空间，4KB的页面大小和一个4字节的PTE，则4KB的页面大小需要12位来表示每个字节，所以还剩下32-12=20位来表示页的虚拟地址，所以一共有2^20个虚拟地址，又因为每个虚拟地址需要4字节的PTE，所以页的大小为2^20*4=4MB。又因为每个进程都有一个独立的页表，且页表常驻于主存中，所以假设有1000个进程，则需要4GB的空间，这是非常兰妃空间的，这里还不考虑64位系统的页表，64位系统会更加复杂。

压缩页表的常用方法是使用层次结构的页表。比如32位虚拟地址空间被分为4KB的页，而每个页表条目都是4字节。假设虚拟地址空间有如下形式：存储器的前2K个页面分配给了代码和数据，接下来的6K个页面还未分配，再接下来1023个页面也未分配，接下来1个页面分配给了用户栈。下图展示了我们为这个虚拟地址空间构造的一个两级的页表层次结构。

一级页表中的每个PTE负责映射虚拟地址空间中一个4MB的片（chunk），这里每一片都是由1024个连续的页面组成。不如PTE0映射第一片，PTE 1映射接下来一片，以此类推。假设地址空间是4GB，则1024个PTE可以覆盖整个空间。（1024*4MB=4GB）。

如果片i中每个页面都未被分配，那么一级PTE i就为空。如果片i中至少有一个页时分配的，那么一级PTE i就指向一个二级页表的基址。二级页表中每个PTE都负责一个4KB的虚拟存储器页面，就像我们查看一级的页表一样。使用4字节的PTE，每个一级和二级页表都是4KB字节，刚好和一个页面大小是一样的。这里其实就是将32位地址划分为10+10+12，12是页内偏移，第一个10位对应于1024个片，第二个10位对应于每个片内1024个页。

二级页表从两方面减少了存储器要求：

  1. 如果一级页表的一个PTE是空的，那么相应的二级页表就根本不会存在，节约了很多空间，因为一个典型的程序，4GB的虚拟地址空间的大部分是未分配的。
  2. 只有一级页表才需要总是在主存中，虚拟存储器系统可以在需要时创建、页面调入或调出二级页表，减少了主存的压力，只有最经常使用的二级页表才需要缓存在主存中。

下图描述了一个使用k级页表层次结构的地址翻译。第k级页表中每个PTE包含某个物理页面的PPN或一个磁盘块的地址。为了构造物理地址，在能够确定PPN之前，MMU必须访问k个PTE。访问k个PTE看起来是非常昂贵的额，但是TLB能够起作用，通过将不同层次上页表的PTE缓存起来，使得带多级页表的地址翻译并不比单级页表慢很多。

一个地址翻译的例子

下面是一个运行在一个有TLB和L1 d-cache的小系统上的地址翻译。有如下性质：

  • 存储器是按字节寻址的
  • 存储器访问是针对一个字节的字的（不是4字节的字）
  • 虚拟地址是14位长的（n=14）
  • 物理地址是12位长的（m=12）
  • 页面大小是64字节的（P=64）
  • TLB是四路组相联的，总共有16个条目
  • L1 d-cache是物理寻址、直接映射的，行大小是4字节，而总共有16个组

下图展示了虚拟地址和物理地址的格式，因为每个页面时2^6=64字节，所有虚拟地址和物理地址的低6位分别作为VPO和PPO，虚拟地址的高8位作为VPN，物理地址的高6位作为PPN。

下图展示了这个小的存储器系统的一个快照，包括TLB，页表的一部分和L1高速缓存。在TLB和高速缓存上面，还展示了访问这些设备时时如何划分虚拟地址和物理地址的位的。

  • TLB：TLB是利用VPN的位进行虚拟寻址的。因为TLB有四个组，所有VPN的低两位就作为组索引（TLBI），VPN剩余高6位作为标记（TLBT），用来区分可能映射到同一个TLB组的不同的VPN。
  • 页表：该页表是单级设计，一共有2^8=256个页表条目，这里图中只给出了前16个。为了方便我们用它的VPN来标识每个PTE，但是事实上这些VPN并不是页表的一部分，也不存储在存储器中。俩每个无效的PTE都用一个破折号表示，即无论这里存储的是什么位值，都是没有任何意义的。
  • 高速缓存：直接映射的高速缓存通过物理地址中的字段来寻址。每个块的大小为4字节，所有物理地址的低2位作为块偏移（CO）。因为有16组，所有接下来4位用来表示组索引（CI）。剩下的6位作为标记（CT）。

例子：当读地址0x03d4处字节的加载指令时，下图给出了该虚拟地址的各个位表示：

MMU从虚拟地址中抽取出VPN(0x0F)，并检查TLB，看它是否因为前面的某个存储器引用，缓存了PTE 0xOF的一个拷贝，TLB从VPN中抽取出TLB索引（0x03）和TLB标记（0x3），组0x03中的第二个条目中有效匹配，所以命中，然后将缓存的PPN（0x0D）返回给MMU。

如果TLB不命中，那么MMU就徐璈从主存中取出相应的PTE。现在MMU有了形成物理地址所需要的所有东西。它通过将来自PTE的PPN（0x0D）和来自虚拟地址VPO（0x14）连接起来，形成物理地址0x354。

接下来，MMU发送物理地址给L1缓存，缓存从物理地址中抽取出缓存偏移CO（0x0），缓存组索引CI(0x5)以及缓存标记CT（0x0D）。如下图所示：

意味组0x5中的标记与CT向匹配，所以缓存命中，读出在偏移量CO处的数据字节0x36，并将它返回给MMU，随后MMU将它返回给CPU。

翻译过程的其他路径也是可能的，比如TLB不命中，那么MMU必须从页表中的PTE中取出PPN，如果得到的PTE是无效的，那么就会产生一个缺页，内核必须调入合适的页面，重新加载这条加载指令。另一种可能是PTE是有效的，但是所需要的存储器块在缓存中不命中。
