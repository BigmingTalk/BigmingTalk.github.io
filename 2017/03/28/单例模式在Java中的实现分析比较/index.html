单例模式在Java中的实现分析比较

Java
  • 单例模式
  • 懒汉
  • 枚举
  • 设计模式
  • 饿汉

本文主要分析单例模式在Java中的各种实现方式，既有线程安全的也有线程不安全的实现，并分析它们各自存在的问题以及性能比较。并结合可能存在的多个类加载器问题、反序列化问题、反射问题、克隆问题做出解答。

单例模式的实现方法

懒汉，线程不安全

这是懒汉模式，所谓懒汉模式就是只有在你需要获取单例对象的时候才去实例化对象，这个模式最大的缺点是线程不安全，之所以线程不安全是因为可能两个线程都判断了singleton==null为true，于是都执行singleton = new Singleton();语句。

懒汉，线程安全

这是上面版本的一个变形，通过在getSingleton方法加入synchronized关键字，确保一个时刻只有一个线程能够进入方法内部。但问题是效率特别低下，因为单例对象只需要创建一次，但是后面可能需要访问很多次，后面每次访问都要对线程进行同步其实没有任何必要的。

饿汉

这是饿汉模式的典型实现，这里通过在类加载的初始化阶段实例化单例对象保证了只有一个单例对象生成，避免了多线程同步的问题。但是问题是饿汉模式没有达到lazy loading的效果，在类加载阶段就实例化单例对象有可能对性能会有较大的影响。

饿汉（变种）

这是上面模式的另一种写法，其实效果和上面是一样的，只不过这里是在static域里面实例化单例对象。

静态内部类

这里与上面两种方法不同的是上面两种在Singleton类初始化的时候就实例化了单例对象，而这里则不是，这里只有在调用getSingleton()方法时才会实例化SingletonHolder对象，从而实例化单例对象singleton，从而起到lazy loading的作用。

这里的实现方式和上面两种方式之所以线程安全是因为JVM在类的初始化阶段JVM会去获取一个锁，这个锁可以同步多个线程对同一个类的初始化。（参考自《Java并发编程的艺术》）。

枚举实现

这是《Effective Java》里面推荐的方式，枚举类型是在JDK 5中才加入的，枚举实现的好处是避免了多线程的同步问题，而且还能防止反序列化重新创建新的对象。不过这种实现方式的坏处就是看起来不是那么直接。

双重锁定模式

这里的关键是singleton定义为volatile的，这里如果singleton不定义为volatile，可能会因为重排序导致导致线程看到一个尚未被初始化的对象，这是因为singleton = new Singleton()这一局代码其实分为3步：

  1. 为对象分配内存空间
  2. 初始化对象
  3. 设置singleton指向刚分配的内存地址

这里第2步和第3步如果singleton不是volatile的，那么可能因为JIT编译器重排序，即这里的第3步先于第2步进行。所以可能造成的结果是线程B看到singleton为null，但是其实已经有另外一个线程A在初始化Singleton对象，从而生成多个Singleton。

通过将singleton定义为volatile类型，可以实现线程安全的延迟初始化，但是这只有在JDK 5或更高版本内才有效（从JDK 5开始使用新的JSR-133里增强了volatile语义）。

单例模式可能存在的问题

类加载器问题

对于任意一个类，都需要由加载它的类加载器和这个类本身一同确立其在Java虚拟机中的唯一性。所以如果用不同的类加载器就能加载出多个类，也就能有多个实例，这些实例操作起来一样，但对JVM来说是不一样的类的实例。例如一些servlet容器对每个servlet使用完全不同的类装载器，这样的话如果有两个servlet访问一个单例类，它们就都会有各自的实例。可以通过使其调用同一个类加载器来解决，解决该问题的方法是

反射问题

反射可以得到私有的构造函数，所以需要在构造函数中加入一个判断，当第二次创建实例时（一般是通过判断对象不为null），则抛出一个运行时错误来解决，当然，枚举类型自动解决了该问题。

克隆问题

从克隆我们可以想到原型模式，原型模式就是通过clone方法实现对象的创建的，clone方式是Object类的方法，每个对象都有。

当我们使用一个单例模式类的对象，调用clone方法，再创建一个新的对象了。某一个对象直接调用clone方法，会抛出异常，即并不能成功克隆一个对象。调用该方法时，必须实现一个Cloneable 接口。这也就是原型模式的实现方式。还有即如果该类实现了Cloneable接口，尽管构造函数是私有的，他也可以创建一个对象。即clone方法是不会调用构造函数的，他是直接从内存中copy内存区域的。所以单例模式的类是不可以实现Cloneable接口的。

总结

单例模式的使用场景主要有两种：

  • 创建某个对象的代价很大，为了避免频繁创建和销毁对象引起的资源浪费，考虑使用单例模式。
  • 该类对象只能有一个，有多个惰性会造成不可预测的错误，比如只有一个缓存等。

对于上面7中方式的写法，第一种方式是线程不安全的，只是示例使用，第二种性能太低，一般也不做考虑，第三种和第四种其实可以说是同一种方式的不同形式，我个人比较推荐枚举方式和静态内部类方式，当然双重锁定方式也很好，不过他们枚举和双重锁定都只能在JDK 5之后才有效，所以个人觉得最好的方式还是静态内部类模式。
